# Общий (супер детальный) промпт для анализа PDF с описанием проекта системы

> **Как использовать:** вставьте этот промпт в чат с ИИ и **прикрепите PDF** (или вставьте текст, извлеченный из PDF).  
> **Цель:** создать полный анализ (продукт + требования + архитектура + риски) и **оценочный бэклог в часах** по дисциплинам (UX/UI, Front-end, Back-end, Интеграция, QA).

---

## ИНСТРУКЦИИ ДЛЯ ИИ

Вы — **Архитектор программного обеспечения + Продуктовый аналитик + Техлид + PO**.  
Проанализируйте предоставленный PDF (описание системного проекта) и подготовьте **крайне детальный**, структурированный и практически полезный ответ.

### Правила и качество
1. **Базируйтесь только на содержимом PDF**. Если что-то не указано явно, **сообщите, что это предположение**, и объясните почему.
2. Перечислите **предположения** отдельно и отметьте их как **(ПРЕДПОЛОЖЕНИЕ)**.
3. Перечислите **вопросы/сомнения** отдельно и отметьте их как **(ВОПРОС)**.
4. Если есть противоречивые фрагменты, зафиксируйте их как **(КОНФЛИКТ)** и предложите решение.
5. Используйте ясный язык с примерами. Для нетехнической аудитории включите краткую версию.
6. Везде, где возможно, создавайте **артефакты**: списки, таблицы, схемы потоков, критерии приемки, матрицу рисков, бэклог, оценки.
7. Если в PDF недостаточно деталей, все равно подготовьте полный результат, указав:
   - то, что можно утверждать с уверенностью,
   - то, что требует определения,
   - и «минимально жизнеспособный» бэклог для валидации проекта (MVP).

## ОЖИДАЕМЫЙ РЕЗУЛЬТАТ (обязательный формат)

### 0) Метаданные анализа
- Название проекта (если есть):
- Версия/дата документа (если есть):
- Автор/организация (если есть):
- Контекст (1–2 предложения):
- Степень полноты PDF (0–100%):
- Краткий список критических пробелов (маркированный список):

---

# 1) Обзор проекта (Что это за проект)
Объясните:
- **Проблема**, которую решает проект
- **Ценностное предложение**
- **Ожидаемые выгоды**
- **Основные функциональные возможности (высокий уровень)**
- **Чем проект НЕ является** (если есть признаки)

Включите:
- Резюме для **неспециалистов** (макс. 10 строк)
- Резюме для **технических специалистов** (макс. 10 строк)

---

# 2) Бизнес-цели и KPI
Идентифицируйте и детализируйте:
- Бизнес-цели (снижение затрат, повышение эффективности, масштабируемость, комплаенс и т.д.)
- Предлагаемые метрики/KPI (даже если они не указаны явно) с:
  - Определением KPI
  - Способом измерения
  - Частотой сбора данных
  - Предлагаемой начальной целью (ПРЕДПОЛОЖЕНИЕ, если необходимо)

Примеры KPI (адаптируйте к контексту PDF):
- Среднее время на операцию
- Снижение количества переделок
- Процент ошибок
- Активное использование (DAU/MAU), если это продукт
- Время отклика / производительность

---

# 3) Целевая аудитория, персоны и роли (RBAC)
Опишите:
- Кто использует систему (внутренние/внешние пользователи)
- Персоны / профили пользователей
- Роли и разрешения (RBAC), включая:
  - Что каждый профиль может видеть/делать
  - Ограничения данных (например, multitenancy, конфиденциальные данные)

**Результат:**
- Таблица: Персона | Цель | Боли | Ключевые функции | Разрешения

---

# 4) Скоуп (В рамках / Вне рамок) + Фазы (MVP vs Будущее)
Подготовьте:
- Что входит **в рамки** проекта (In Scope)
- Что находится **вне рамок** проекта (Out of Scope)
- Разделение по фазам:
  - **MVP (Фаза 1)**
  - **Фаза 2**
  - **Фаза 3 / Будущая дорожная карта**

**Результат:**
- Список, приоритезированный по влиянию x усилиям (если возможно)
- Указание зависимостей/рисков

---

# 5) Дидактическая аналогия (для быстрого понимания)
Создайте **2–3 аналогии**, чтобы объяснить систему для:
1) нетехнического специалиста  
2) технической команды  
3) бизнес-стейкхолдера  

Каждая аналогия должна содержать:
- «Это как…»
- Сопоставление элементов (например, «пользователь = клиент», «поток = конвейер» и т. д.)
- Ограничения аналогии (где она перестает работать)

---

# 6) Пути и потоки системы (User Flows)
Извлеките и/или выведите:
- Основные потоки (happy path)
- Альтернативные потоки
- Потоки исключений/ошибок
- Состояния системы (например, «черновик», «опубликовано», «отменено»)

**Обязательный результат:**
- Список потоков с пронумерованными шагами
- Для каждого потока: вход, выход, правила, валидации, сообщения пользователю
- Если возможно: текстовая псевдо-диаграмма (например, блок Mermaid — опционально)

---

# 7) Функциональные требования (RF)
Перечислите и детализируйте:
- Требования по модулям
- Связанные бизнес-правила
- Валидации
- Разрешения
- Ожидаемые результаты

**Результат:**
- Таблица: ID | Требование | Описание | Приоритет (MoSCoW) | Зависимости | Примечания

---

# 8) Нефункциональные требования (RNF) — обязательно
Идентифицируйте и предложите (если нет в PDF):
- Производительность (p95, p99, пропускная способность, задержка)
- Масштабируемость (горизонтальная/вертикальная)
- Доступность (SLA)
- Безопасность (аутентификация, авторизация, OWASP, шифрование, комплаенс/GDPR)
- Наблюдаемость (логи, метрики, трассировка)
- Аудит (журнал аудита)
- Юзабилити (доступность, адаптивность)
- Совместимость (браузеры, мобильные устройства и т. д.)
- Поддерживаемость и тестируемость
- Бэкап/восстановление и хранение данных
- Интернационализация/локаль (если применимо)

**Результат:**
- Таблица: Категория | Требование | Как валидировать/тестировать | Уровень (MVP/Будущее) | Примечания

---

# 9) Структура системы (как она будет устроена)
Опишите четко:
- Видение модулей (например, Админка, Операции, Отчеты, Настройки, Интеграции)
- Компоненты (фронт, бэк, база данных, кэш, очереди, хранилище)
- Границы контекста (Bounded Contexts), если это уместно
- Внутренние зависимости

**Результат:**
- Карта модулей + зоны ответственности
- Список внутренних/внешних API по модулям (если применимо)

---

# 10) Техническая архитектура (предлагаемая или неявная)
Объясните:
- Архитектурный стиль (модульный монолит, микросервисы, event-driven и т. д.)
- Паттерны (Clean Architecture, DDD, CQRS и т. д.)
- Коммуникация (REST, gRPC, месседжинг)
- Стратегия данных (SQL/NoSQL, кэш, индексы)
- Стратегия состояния (stateless, сессии)
- Стратегия развертывания (контейнеры, CI/CD)

**Результат:**
- Текстовая схема архитектуры (компоненты и связи)
- Список архитектурных решений с плюсами/минусами (краткий ADR)

---

# 11) Домен и модель данных (Сущности)
Извлеките или предложите:
- Основные сущности
- Релевантные поля
- Связи
- Правила целостности
- Перечисления (Enums)/статусы

**Результат:**
- Словарь данных (таблица по каждой сущности)
- Бизнес-правила для каждой сущности
- Предложения по таблицам/индексам (если уместно)

---

# 12) Внешние интеграции и контракты
Опишите:
- Внешние системы (ERP, шлюзы, аутентификация и т. д.)
- Тип интеграции (REST/SOAP, webhook, очередь)
- Аутентификация/безопасность интеграции
- Передаваемые данные (вход/выход)
- Ошибки и повторные попытки
- SLA и лимиты (rate limit)

**Результат:**
- Таблица: Интеграция | Цель | Метод | Auth | Payload | Ошибки | Примечания

---

# 13) Аутентификация, авторизация и комплаенс (GDPR)
Детализируйте:
- Логин/SSO (если есть)
- RBAC/ABAC (правила доступа)
- Конфиденциальные данные и политика доступа
- GDPR: правовая основа, согласие, хранение, анонимизация (если применимо)

**Результат:**
- Матрица: Профиль x Разрешение x Модуль
- Список рисков GDPR и способы минимизации

---

# 14) Аудит, логи и прослеживаемость
Определите:
- Что должно аудироваться (критические действия)
- Минимальная структура записи (кто, когда, что, до/после)
- Требования к хранению и поиску
- Наблюдаемость (структурированные логи, метрики, трассировка)

**Результат:**
- Чек-лист аудируемых событий
- Пример события аудита (иллюстративный JSON)

---

# 15) Критерии приемки и Definition of Done
Для основных требований подготовьте:
- Критерии приемки (Given/When/Then)
- Definition of Done (DoD) по типам задач (UX/FE/BE/QA)

**Результат:**
- Список критериев для ключевых требований
- Чек-лист DoD

---

# 16) Препятствия, замечания и зависимости
Извлеките:
- Явные препятствия из PDF
- Технические/бизнес-зависимости
- Ограничения (срок, технология, наследие, команда, комплаенс)

**Результат:**
- Приоритезированный список (по влиянию)
- Предложенные меры по минимизации

---

# 17) Риски проекта (матрица)
Создайте матрицу:
- Риск
- Вероятность (Низкая/Средняя/Высокая)
- Влияние (Низкое/Среднее/Высокое)
- Критичность (Влияние x Вероятность)
- Меры по минимизации
- План на случай реализации риска

---

# 18) Бэклог задач (супер детальный) + оценка в часах по дисциплинам
Создайте бэклог, организованный по эпикам и историям:
- **Эпики**
- **Фичи**
- **User Stories** (формат: «Как [персона], я хочу [действие], чтобы [выгода]»)
- **Технические задачи** (FE/BE/Инфра)
- **Критерии приемки**
- **Зависимости**
- **Риски**
- **Оценка в часах**, разделенная по:
  - UX/UI
  - Front-end
  - Back-end
  - Интеграция (между фронтом и бэком / контракты / правки)
  - QA

### Правила оценки
- Укажите, является ли оценка **консервативной** или **агрессивной**
- Всегда указывайте предположения (ПРЕДПОЛОЖЕНИЕ)
- Разделите MVP vs Будущее (если возможно)
- Идентифицируйте элементы, требующие **spike** (технического исследования)

**Обязательный результат:**
1) **Таблица бэклога**  
   - ID | Эпик | История/Задача | Описание | Зависимости | Критерии приемки | UX | FE | BE | INT | QA | Итого | Примечания

2) **Итого по дисциплинам**  
   - UX всего (ч)
   - Front-end всего (ч)
   - Back-end всего (ч)
   - Интеграция всего (ч)
   - QA всего (ч)
   - Общий итог (ч)

3) **Итого по фазам**
   - MVP (ч) + список элементов
   - Фаза 2 (ч) + список элементов
   - Фаза 3 (ч) + список элементов

---

# 19) План релизов и стратегия валидации
Предложите:
- Релиз 0 (discovery): прототип + валидация
- Релиз 1 (MVP): минимальная ценность
- Релиз 2+: расширение

Включите:
- Что измерять в каждом релизе (KPI)
- Как собирать обратную связь (аналитика, интервью, логи)

---

# 20) Финальные вопросы (критические уточнения)
Составьте список из **10–20 самых важных вопросов** для уточнения скоупа и снижения рисков, например:
- Какой SLA требуется?
- Какие интеграции обязательны для MVP?
- Кто утверждает правила/потоки?
- Требуется ли multitenancy?
- Какие данные подпадают под GDPR?
- Ожидаемый объем пользователей/запросов?

## ФОРМАТ ПРЕДСТАВЛЕНИЯ
- Используйте **нумерованные заголовки** точно так же, как выше.
- Используйте таблицы там, где это указано.
- Будьте детальными, но лаконичными: делайте акцент на том, что влияет на принятие решений.
- Если в документе есть сложные термины, создайте **глоссарий** в конце.
